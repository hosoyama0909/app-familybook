# 支援プロセス：SUP.8 構成管理 / SUP.9 問題解決 / SUP.10 変更依頼

## SUP.8 構成管理（＝Git 運用）

### ブランチ戦略
- `main`：リリース可能な状態。
- 開発ブランチ：`claude/<topic>`（機能単位）。完了後にレビュー→merge。

### コミット規約
- 1コミット＝意味のある1単位。メッセージに**関連要求ID/CR番号**を含める。
- 例：`Stop2: ヒストリーマップ追加 (SWR-MAP-01, CR-12)`

### 構成識別（何をバージョン管理するか）
| 構成品目 | 場所 |
|----------|------|
| 実装 | `03_implementation/index.html` |
| 要求・設計・テスト仕様 | 各 `0X_*` フォルダ（00_process〜06_management） |
| テストコード | `04_test/**` |
| リリース記録 | `/CHANGELOG.md`（新設）＋ Git タグ |

### ベースライン
- 各 Stop 完了時に Git タグを打つ（例：`v0.1-stop1`）。タグ＝トレース可能なベースライン。

## SUP.9 問題解決管理（＝バグ管理）

- 不具合は GitHub Issue に `bug` ラベルで起票。テンプレ項目：
  **再現手順 / 期待 / 実際 / 影響要求ID / 重大度**。
- 修正は該当 SWR/テストに紐付け、回帰テスト（QTC）を追加してから close。
- 重大度：S1(データ消失/起動不可) 〜 S3(軽微)。S1は最優先。

## SUP.10 変更依頼管理（＝Change Request）

- 機能追加・仕様変更は Issue に `change-request` ラベルで起票（＝CR）。
- CR 承認 → 影響分析（どの STK/SYR/SWR に波及するか）→ 要求更新 → 実装 → テスト →
  **トレーサビリティ更新** → PR レビュー → merge。
- CR番号は Issue 番号を流用（例：CR-12 = #12）。
